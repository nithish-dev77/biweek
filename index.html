Offline-First Mobile Apps

Caching, Local Storage, & Syncing in React Native

A simple guide and presentation for building fast, reliable mobile apps using Expo and React Native.

Target: Engineering Team
02. Introduction

Why Build Offline-First?

Offline-first means designing your app so it always works perfectly, even without the internet. It uses data saved on the phone to make everything feel fast and smooth.

⚡

Instant Speed

The app loads data from the phone first. Screens show up instantly without waiting for the internet.

🛡️

Always Works

The app works perfectly in subways, elevators, or places with bad signal. No more annoying loading spinners.

🔋

Saves Battery

Using local data and grouping network updates saves battery life and uses less cellular data.

03. Architecture

The Core Idea

Old Way: Cloud-First

  • ✕ App asks internet for data.
  • ✕ App waits (loading spinner).
  • ✕ App shows data or an error.
  • The online database is the boss.
VS

New Way: Offline-First

  • ✓ App reads/saves to phone's storage.
  • ✓ App shows data immediately.
  • ✓ Phone syncs with internet in the background.
  • The phone's storage is the boss.
04. Tooling

Expo & React Native Tools

Expo gives us great tools for offline apps. We use these along with community tools to build a strong system.

Saving & Fetching Data

  • Zustand/Redux Persist - Saves app settings.
05. Storage

Local Storage Options

Picking the right local database is very important. It decides how much data you can save and how fast the app runs.

Tool Type Speed Best For
AsyncStorage Simple Text Slow Login tokens, old apps.
react-native-mmkv Simple Data Super Fast Saving internet data, app settings, fast updates.
expo-sqlite Tables & Relations Fast Complex searching, lots of data, offline features.

07. Database Data

Expo SQLite Basics

For complex apps, we need a real database. expo-sqlite lets us talk to the database instantly, without waiting for the old React Native bridge.

import * as SQLite from 'expo-sqlite'; // Open database instantly const db = SQLite.openDatabaseSync('myApp.db'); // Create a table safely db.execSync(` CREATE TABLE IF NOT EXISTS sync_queue ( id INTEGER PRIMARY KEY AUTOINCREMENT, action TEXT NOT NULL, payload TEXT NOT NULL, timestamp DATETIME DEFAULT CURRENT_TIMESTAMP ); `); // Prepare and run a save action instantly const statement = db.prepareSync('INSERT INTO sync_queue (action, payload) VALUES (?, ?)'); statement.executeSync(['CREATE_USER', JSON.stringify({name: 'Alice'})]);
08. Simple Saving

MMKV: Super Fast Saving

When you don't need a complex database, but just want raw speed for saving simple data or settings, MMKV is the best choice.

10. Data Flow

How to Get Data

An offline app needs clear rules for how it gets data. We use different methods based on what the app needs.

Cache First

Check the phone first. If data is there, use it right away. Best for data that rarely changes.

Network First

Try the internet first. If it fails, use the phone's saved data. Best for data that changes a lot.

Show Old, Fetch New

Show saved data instantly. Meanwhile, get new data from the internet in the background. Gives the best user experience.

12. Implementation

Writing the Code

Tools like React Query handle the complex "Show Old, Fetch New" logic for us easily.

import { useQuery } from '@tanstack/react-query'; import { storage } from './mmkv'; const fetchUser = async () => { const response = await fetch('/api/user'); return response.json(); }; export function UserProfile() { const { data, isFetching } = useQuery({ queryKey: ['user'], queryFn: fetchUser, staleTime: 1000 * 60 * 5, // Data is considered "fresh" for 5 minutes // React Query instantly returns the saved data, // while checking for updates in the background. }); return ( <View> <Text>{data?.name}</Text> {isFetching && <Text>Updating in background...</Text>} </View> ); }
13. Handling Writes

Saving Changes Offline

Loading data offline is easy. But saving user changes while offline is harder. When a user creates something offline, the app must:

  • Accept the user's input without showing an error.
  • Create a temporary ID right there on the phone.
  • Update the phone's database right away.
  • Save the action in an Offline Queue to send to the internet later.
Warning: Don't rely on the server to create IDs if you want the app to work offline. Always create IDs on the phone first.

15. User Experience

Instant Screen Updates

Also known as "Optimistic UI," this means updating the screen instantly when the user clicks something, assuming the internet request will work out fine later.

The Old Way
  1. User clicks "Like"
  2. App shows loading spinner
  3. ... sending to internet ... (wait 1 second)
  4. Internet says "Success!"
  5. Button turns Blue
The Instant Way
  1. User clicks "Like"
  2. Button turns Blue RIGHT AWAY
  3. ... internet request runs silently in background ...
  4. Internet says "Success!" (nobody notices)
  5. If it fails: Show an error and turn button gray again
16. Connectivity

Sending Saved Tasks

When do we send our saved tasks to the internet? We listen for when the internet comes back or when the user opens the app.

import NetInfo from '@react-native-community/netinfo'; import { AppState } from 'react-native'; // 1. Listen for the internet turning back on NetInfo.addEventListener(state => { if (state.isConnected && state.isInternetReachable) { sendSavedTasks(); } }); // 2. Listen for the user opening the app AppState.addEventListener('change', nextAppState => { if (nextAppState === 'active') { sendSavedTasks(); } });
17. Resolving Problems

Understanding Sync Conflicts

In an offline app, two users might edit the exact same thing at the same time. When they both connect to the internet, their changes might fight each other. This is a sync conflict.

The Problem

1. Start: Document title is "Draft"
2. User A (Offline) changes title to "Final Draft" at 10:00 AM
3. User B (Online) changes title to "Published" at 10:05 AM & saves
4. User A goes online at 10:10 AM & tries to save "Final Draft"

20. Conclusion

Summary & Best Practices

Do's

  • • Create IDs on the phone, not the server.
  • • Treat the phone's storage as the main boss.
  • • Use good databases (like SQLite) for big apps.
  • • Always update the screen instantly for user clicks.
  • • Hide deleted items instead of erasing them completely from the phone.

Don'ts

  • • Don't freeze the screen waiting for the internet.
  • • Don't use AsyncStorage for large lists of data.
  • • Don't rely on the server to number your items (like 1, 2, 3).
  • • Don't build complex sync systems from scratch if good tools exist.
